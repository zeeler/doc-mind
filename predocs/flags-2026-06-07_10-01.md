# 资料管理改造方案

> 日期: 2026-06-07 设计 → 2026-06-21 全部完成 | 状态: ✅ 已实施

## 概述

将现有"文档管理"模块升级为"资料管理"，支持更广泛的资料类型和智能标签。

## 四项改造

| # | 改造项 | 说明 | 状态 |
|---|--------|------|------|
| 1 | 去掉集合（Collection） | 移除集合模型、路由、文档关联、前端 UI | ✅ |
| 2 | 自动打标签 | 资料入库后 LLM 自动分析内容并打标签；每条资料增加"重新打标签"按钮 | ✅ |
| 3 | URL 导入 | 用户粘贴 URL，系统抓取网页内容并作为资料入库 | ✅ |
| 4 | Chrome 书签导入 | 上传 Chrome 导出的书签 HTML 文件，批量导入其中所有链接 | ✅ |

---

## 一、去掉集合

### 现状

`Collection` 模型通过 `collection_documents` 多对多表关联 `Document`，提供分组功能。

涉及删除的文件/代码：

| 位置 | 操作 |
|------|------|
| `server/models/collection.py` | 删除整个文件 |
| `server/models/document.py` | 删除 `collections` relationship 和 `collection_documents` 导入 |
| `server/routers/collections.py` | 删除整个文件 |
| `server/routers/documents.py` | 删除 `collection` 查询参数、`collect` batch action、`add_collections`/`remove_collections` in PUT |
| `server/main.py` | 删除 `collections_router` 注册 |
| `server/database.py` | `_migrate()` 中删除 `collections` 和 `collection_documents` 建表语句 |
| `server/templates/index.html` | 删除集合相关 UI 组件 |

### 建议

不做数据迁移（MVP 阶段，存量 collection 数据可直接丢弃）。`_migrate()` 中保留 `CREATE TABLE IF NOT EXISTS` 的旧表不删除（不影响运行），仅停止新代码引用。

## 二、自动打标签

### 现状

标签（Tag）已存在，但目前仅支持手动添加（上传时传入、批量操作中指定）。

### 方案

#### 自动打标签时机

在资料处理完成（`status=done`）后触发。具体位置：worker 的 `_execute_job` 中 `full_index` 完成后、`create_jobs_for_document` 创建的任务链中增加 `auto_tag` 类型，或直接在 `full_index` 完成后同步调用。

**推荐方案**：在 `full_index` 完成后同步调用（标签生成依赖已索引的文本内容，同步执行可确保标签在资料 status=done 时已就绪）。

#### 标签生成方式

使用 LLM 分析资料摘要（title + 前 3 个 chunk 的前 200 字）生成 3-5 个标签。Prompt 设计：

```
根据以下资料内容，生成 3-5 个简短的分类标签（每个标签 2-6 个中文字）。
标签应反映资料的主题领域，例如"机器学习"、"项目管理"、"财报分析"等。
只输出标签名，一行一个，不要编号或其他文字。

资料标题：{title}
内容摘要：
{summary}
```

#### 标签存储

生成的标签名经过 `_normalize_tag_name` 标准化后，通过 `_get_or_create_tag` 获取或创建，关联到 Document。**覆盖式打标签**：新标签完全替换旧标签（而非追加），确保重新打标签时不会残留旧标签。

#### "重新打标签"按钮

- **前端**：每条资料行增加一个标签图标按钮（`Tag` icon），hover 提示"重新打标签"
- **后端**：`POST /api/v1/documents/{doc_id}/retag`
- **行为**：清除该资料现有标签 → 重新调用 LLM 生成标签 → 返回新标签列表
- **状态**：该操作同步执行（标签生成通常 1-3 秒），前端显示 loading 状态

#### 新增/修改文件

| 位置 | 操作 |
|------|------|
| `server/services/auto_tagger.py` | 新增 — `AutoTagger` 类，封装 LLM 标签生成逻辑 |
| `server/services/worker.py` | 修改 — `full_index` 完成后调用 auto-tag |
| `server/routers/documents.py` | 修改 — 新增 `POST /{doc_id}/retag` 端点 |
| `server/config.py` | 修改 — 新增 `auto_tag_enabled: "true"` 配置项 |
| `server/templates/index.html` | 修改 — 增加标签图标按钮 + retag 交互 |

## 三、URL 导入

### 方案

新增端点 `POST /api/v1/documents/import-url`，接收 JSON body：

```json
{
  "url": "https://example.com/article",
  "folder_path": ""
}
```

### 处理流程

```
接收 URL
  → 校验 URL 格式（http/https）
  → httpx 抓取网页内容（timeout=30s, follow_redirects=true）
  → 用 BeautifulSoup 提取正文（去除 script/style/nav/footer，保留 article/main/body 中的文本）
  → 提取 <title> 作为资料标题
  → 将提取的纯文本保存为 .md 文件到 data/files/{doc_id}/
  → 创建 Document 记录（file_type="url", checksum=url 的 SHA256）
  → 创建 quick_scan + full_index 任务
  → 返回资料 ID
```

### URL 去重

对 URL 字符串做 SHA256 作为 checksum，复用现有文件去重逻辑。同一 URL 重复导入时返回已有资料（与文件上传去重行为一致）。

### 内容提取策略

使用 `bs4` + `markdownify` 或手工提取：

1. 优先使用 `<article>` 标签内容
2. 其次使用 `role="main"` 或 `.content` / `.post-content` 等常见 class
3. 兜底使用 `<body>` 全文
4. 移除 `<script>`, `<style>`, `<nav>`, `<footer>`, `<header>` 标签
5. 用 `soup.get_text(separator="\n")` 提取纯文本，保留段落结构

### 新增/修改文件

| 位置 | 操作 |
|------|------|
| `server/services/url_fetcher.py` | 新增 — URL 抓取 + 正文提取 |
| `server/routers/documents.py` | 修改 — 新增 `POST /import-url` 端点 |
| `server/models/document.py` | 修改 — `file_type` 增加 `"url"` |
| `server/services/parser.py` | 修改 — `SUPPORTED_TYPES` 或 upload 流程适配 url 类型 |
| `pyproject.toml` | 修改 — 新增 `beautifulsoup4`, `lxml`（已有 bs4） |

**注意**：`requirements.txt` 已有 `beautifulsoup4>=4.12.0`，无需新增依赖。如需更好的 HTML 解析可选加 `lxml`。

## 四、Chrome 书签导入

### Chrome 书签格式

Chrome 导出书签生成标准 Netscape Bookmark File Format (.html)：

```html
<!DOCTYPE NETSCAPE-Bookmark-file-1>
<META HTTP-EQUIV="Content-Type" CONTENT="text/html; charset=UTF-8">
<TITLE>Bookmarks</TITLE>
<H1>Bookmarks</H1>
<DL><p>
    <DT><H3>书签栏</H3>
    <DL><p>
        <DT><A HREF="https://example.com" ADD_DATE="1718323200">Example</A>
        <DT><H3>子文件夹</H3>
        <DL><p>
            <DT><A HREF="https://other.com">Other Site</A>
        </DL><p>
    </DL><p>
</DL><p>
```

### 方案

新增端点 `POST /api/v1/documents/import-bookmarks`，接收上传的 `.html` 书签文件。

### 处理流程

```
接收书签 HTML 文件
  → 用 BeautifulSoup 解析 <DT><A HREF="..."> 标签
  → 提取所有书签链接及标题、文件夹层级
  → 按文件夹层级分组，生成导入预览列表
  → 返回预览给前端（不立即导入）
  → 前端确认后，逐条或批量调用 /import-url 导入
```

### 两阶段设计

**阶段 1 — 预览**（`POST /import-bookmarks?preview=true`）：

返回解析结果，不做实际导入：

```json
{
  "code": "OK",
  "data": {
    "total": 42,
    "folders": [
      {
        "name": "书签栏/技术",
        "bookmarks": [
          {"title": "Example", "url": "https://example.com", "add_date": "2024-06-01"},
          ...
        ]
      }
    ]
  }
}
```

**阶段 2 — 确认导入**（`POST /import-bookmarks`）：

前端传入要导入的 URL 列表（支持全选或按文件夹勾选），后端异步逐条导入：

```json
{
  "urls": ["https://example.com", "https://other.com"],
  "folder_path": "书签导入/技术"   // 可选，将导入的资料归入此文件夹
}
```

返回任务 ID 或直接返回导入数量。

### 文件夹映射

书签的文件夹层级映射到现有 `folder_path` 字段，如书签在"书签栏 > 技术 > AI"下，导入后 `folder_path = "书签导入/技术/AI"`。

### 新增/修改文件

| 位置 | 操作 |
|------|------|
| `server/services/bookmark_parser.py` | 新增 — 解析 Chrome 书签 HTML |
| `server/routers/documents.py` | 修改 — 新增 `POST /import-bookmarks` 端点 |

## 待讨论

1. **自动打标签的 LLM 成本**：每条资料入库都调用一次 LLM。是否需要可配置关闭？默认建议开启（单次调用 token 量很小 ~200 input + ~50 output）。
2. **URL 导入的正文提取质量**：不同网站 HTML 结构差异大，正文提取可能不完整。是否接受"尽力而为"的策略，还是需要更高精度的方案（如 `trafilatura` 库）？
3. **书签导入的去重策略**：如果某 URL 已通过其他方式导入（如直接 URL 导入），书签导入时是跳过还是创建新记录？建议复用 URL 的 SHA256 去重。
4. **书签导入量级**：单次导入可能有数百条书签。是需要后台任务队列逐条处理，还是预期前端等待？建议批量异步处理，返回 job 统计。
5. **前端改动范围**：去掉集合后，左侧导航栏和资料列表筛选是否需要其他替代功能？目前 `folder_path` 和 `tag` 已提供分类能力。
