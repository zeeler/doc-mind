## Bug

### 1. `build_qa_prompt` 的 `memory_context` 参数是死参数

**文件**: [server/services/rag.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/rag.py:68-75)

```python
def build_qa_prompt(
    question: str,
    chunks: list[dict],
    memory_context: str = "",
    web_sourced: bool = False,
) -> str:
```

函数体内部三个分支（空 chunk、KB prompt、web prompt）都不引用 `memory_context`。调用方 `ask_sync` 传了这个参数但没有使用它的返回值中的 memory 部分——memory 是通过外层的 `system_parts.append(memory_context)` 注入的。`ask_stream` 则根本没传 `memory_context` 给 `build_qa_prompt`。

两个调用方行为不一致：`ask_sync` 多传了一个无用的参数，`ask_stream` 正确跳过了它。不影响功能，但让人困惑。

**建议**：从 `build_qa_prompt` 签名中移除 `memory_context` 参数，并从 `ask_sync` 的调用中去掉它。

### 2. `_update_index` 计数方法会错误统计记忆内容中的 markdown 列表项

**文件**: [server/services/memory_md_exporter.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/memory_md_exporter.py:110-111)

```python
count = len([l for l in content.split("\n") if l.startswith("- ")])
```

如果记忆内容本身包含 markdown bullet list（比如从对话中记录了一段带 `- item` 格式的摘要），这些行也会被计入总数，导致索引中的"共 N 条"数字偏大。

**建议**：改为统计 `"- ["` 开头的行（记忆项以 `- [类型]` 开头），或统计 `"- "` 但排除以 `- [` 以外的行。

## 性能问题

### 3. 每次 `/ask` 在后台线程中创建新的 `LLMAdapter`

**文件**: [server/routers/chat.py](/Users/terry/Documents/cc_projects/my_agent1/server/routers/chat.py:133-134)

```python
def _observe_bg():
    ...
    mem_mgr = MemoryManager(config=cfg, llm=LLMAdapter(cfg))
```

`LLMAdapter.__init__` 会创建 `OpenAI` / `AsyncOpenAI` client 实例。每轮对话都新建一个，在高频使用下会产生不必要的 socket 资源和 GC 压力。同样的问题也存在于 [routers/memories.py](/Users/terry/Documents/cc_projects/my_agent1/server/routers/memories.py:12) 的 `_get_mgr()` —— 为 `list_memories_endpoint`（不需要 LLM）也创建了 LLMAdapter。

**建议**：将 `LLMAdapter` 实例缓存为模块级单例，与 config 联动失效。或者在 `MemoryManager.observe()` 中再惰性创建 LLM 实例。

### 4. `consolidate()` 对每条记忆做一次 ChromaDB `query()`

**文件**: [server/services/memory_manager.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/memory_manager.py:190-197)

```python
all_mems = self.store.get_all(limit=10000)
for mem in all_mems:
    similar = self.store.search(mem["content"], top_k=3)
```

虽然是 O(n) 而非 O(n²)，已经比设计文档好了很多。但 500 条记忆就是 500 次 ChromaDB 查询（每次有文件 I/O），在单线程中顺序执行可能耗时数十秒。没有进度回调或超时保护，长时间阻塞 HTTP 请求。

**建议**：加一个 `max_memories` 参数限制单次处理的记忆数（默认 500），超出的分批执行。或者用 `ThreadPoolExecutor` 并行查询。

## 稳健性

### 5. `memorize()` 不校验 `scope` 参数

**文件**: [server/services/memory_manager.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/memory_manager.py:117)

```python
def memorize(self, content: str, mem_type: str = "manual",
             scope: str = "global", metadata: dict | None = None) -> str:
```

传 `scope="invalid"` 会原样存入，但 `search()` 的 scope 过滤只识别 `"global"` 和 `"session"`，这些记忆将永远无法被检索到。`mem_type` 也没有校验（虽然 `observe()` 内部有校验）。

**建议**：在 `memorize()` 入口处校验 `scope in ("global", "session")`，非法值直接抛 `ValueError`。

### 6. `recall()` 对 `collection.get()` 产生的记忆没有 `score` 字段

**文件**: [server/services/memory_manager.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/memory_manager.py:72-81)

```python
candidates = []
for m in global_mems + session_mems:
    if m["id"] not in seen:
        seen.add(m["id"])
        candidates.append(m)
```

`store.search()` 返回的记忆有 `score` 字段（余弦相似度），这没问题。但如果有任何路径通过 `store.get_all()` 获取记忆再加入 candidates，这些记忆就没有 `score` 字段，`recall()` 中的 `m["score"]` 会抛 `KeyError`。当前代码不会触发此路径，但注释/重构时容易踩坑。

## 代码质量

### 7. `_get_manager()` 用 `llm=None` 创建单例，与 `summarize_conversation()` 的行为不一致

**文件**: [server/services/memory.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/memory.py:19-27)

```python
def _get_manager() -> MemoryManager:
    ...
    _manager = MemoryManager(config=config, llm=None, ...)
```

vs

```python
def summarize_conversation(conv_id: str) -> int:
    ...
    mgr = MemoryManager(config=config, llm=llm, ...)  # 不走单例
```

`summarize_conversation` 不走单例，自己创建了一个带 LLM 的 MemoryManager。功能上没问题（因为需要 LLM），但同一个 ChromaDB 目录有两个 MemoryManager 实例各自创建 MemoryStore（底层 ChromaDB client 是共享的）。这是无害的重复，但代码路径不一致。

**建议**：给 `_get_manager()` 加一个可选的 `llm` 参数，让 `summarize_conversation` 也复用单例。

## 总结

实施质量扎实，之前审查中提出的严重问题基本全部解决。7 个发现均为轻度问题（1 个 Bug、2 个性能、3 个稳健性/代码质量），没有阻塞性问题。

唯一遗留的设计缺口是 #7（会话结束兜底逻辑仍未实现）——`observe()` 只按间隔触发，不会在会话结束时主动做最后的摘要。如果用户在一次长对话中途关闭了标签页，最后几轮对话的记忆可能永久丢失。
