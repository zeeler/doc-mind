### 2. `_execute_job` 跨 session 操作导致日志显示的 chunk 数量不准确

**文件**: [server/services/worker.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/worker.py:112-124)

```python
with get_session_ctx() as s:
    doc = s.get(Document, job.document_id)  # chunk_count = 0
    ...
    index_document(job.document_id, text, config)  # 内部 session 写入 chunk_count
    doc.status = "done"
    ...
    s.commit()
logger.info(f"全文索引完成: {doc.title} ({doc.chunk_count} chunks)")  # 永远显示 0
```

`index_document()` 打开独立的 session 写入 `chunk_count`，外层 session 的 `doc` 对象不会自动刷新（因为 `expire_on_commit=False`），导致日志中 `doc.chunk_count` 永远为 0。虽然不影响数据库正确性，但运维日志完全不可用。

**建议**：在 `index_document()` 返回后执行 `s.refresh(doc)` 刷新对象状态。

### 4. `_is_web_search_needed` 的评分阈值基于错误的假设

**文件**: [server/services/rag.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/rag.py:86-94)

```python
avg = sum(c.get("score", 0.0) for c in chunks) / len(chunks)
if avg < 0.006:
    return True
good = [c for c in chunks if c.get("score", 0.0) > 0.01]
if len(good) < 2 and len(chunks) < 3:
    return True
```

RRF 融合分数的实际范围约为 0.008–0.017（基于 k=60 的 RRF 公式）。`avg < 0.006` 这个阈值过于宽松：任何有向量搜索参与的混合搜索几乎永远不会触发此分支。该阈值看起来像是为余弦相似度（范围 0–1）设计的。在纯 FTS5 模式（无 embedding 模型）下，`score` 退化为 `1.0/(1+i)`（rank 1 = 0.5, rank 10 = 0.09），阈值也基本不生效。

**建议**：根据 RRF 分数重新校准阈值，或为不同 match_type 设置不同的阈值。

**文件**: [server/services/rag.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/rag.py:56-73)

```python
def _build_kb_prompt(question, chunks, memory_section, history_section=""):
```

### 6. `hybrid_search` 中 MMR 重排和 TOC/junk 过滤的顺序问题

**文件**: [server/services/search.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/search.py:189-207)

```python
merged = self._mmr_rerank(candidate_pool, query, config, k, lambda_val)  # 先 MMR
...
```

MMR 可能选出目录 chunk 或垃圾内容 chunk，然后被后续过滤移除，导致最终结果少于 k 条。

### 7. `MemoryStore.search` 的 distance→score 转换对非余弦距离不适用

**文件**: [server/services/memory_store.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/memory_store.py:29)

```python
"score": 1.0 - distances_list[i] if i < len(distances_list) else 0.0,
```

ChromaDB `query()` 的 `distances` 依赖于 collection 的 `hnsw:space` 参数（默认为 l2）。对于 L2 距离，值可能远大于 1.0，`1.0 - distance` 会变成负数。这与 `add_memory` 中 `MEMORY_DEDUP_THRESHOLD = 0.85` 的去重阈值产生冲突：如果距离 > 1.0（L2 空间下很常见），score 就是负数，永远无法触发合并去重。

**建议**：在创建 collection 时显式指定 `hnsw:space: "cosine"`，或对非余弦距离做归一化处理。

### 8. `_expand_query` 中正则对不同提问句式匹配不完整

**文件**: [server/services/retriever.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/retriever.py:54-62)

```python
m = re.match(r"(总结|概述|介绍|讲讲)(.+?)的?(要点|内容|方面)?$", query)
```

`(.+?)` 非贪婪匹配 + 可选的 `的?` 导致"总结一下这本书的内容"匹配到 topic = "一下这本书的内容"（"一下"也被吞进去了）。像"这本书讲了什么"这种常见句式则完全不会命中该模式。

**建议**：在模式前添加常见的填充词清理步骤（如去掉"一下"、"帮我"、"来说"等），再补充"讲了什么"类句式。

### 10. `pipeline.py` 的 `process_document` 是死代码

**文件**: [server/services/pipeline.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/pipeline.py:133-195)

`process_document` 函数执行"解析 → 切块 → embedding → 写入"的完整流程，但整个代码库中没有任何地方调用它。Worker 使用的是 `index_document`（只切块和索引），而 `process_document` 相比 `index_document` 多了内部的 `parse_file` 调用和 Markdown 备份逻辑。这可能是重构遗留。

**建议**：要么删除，要么在文档重处理场景下使用它替代当前 worker 中的手动 `parse_file` + `index_document` 分步调用。

### 14. `stale_recovery` 线程和 `_recover_stuck_jobs` 存在功能重叠

**文件**: [server/services/worker.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/worker.py:33) + [server/routers/jobs.py](/Users/terry/Documents/cc_projects/my_agent1/server/routers/jobs.py:55)

启动时 `_recover_stuck_jobs` 会恢复 running job，而 `start_stale_recovery` 又启动了定期后台恢复。两者逻辑一致，但缺少协调——如果 stale recovery 在启动后立即触发，可能与初始恢复产生竞态。

### 15. `Embedder` 每次查询都重建

**文件**: [server/services/search.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/search.py:110-117)

`_get_query_embedding` 每次调用都创建新的 `Embedder()` 实例，而 `Embedder` 初始化时会创建 `LLMAdapter`（含 OpenAI client）。对于高频查询场景，这会产生不必要的 GC 压力。

**建议**：在 `SearchService` 上缓存 `Embedder` 实例。

## 架构观察

- **Worker 的 Job 认领锁覆盖了整个 commit**（[worker.py](/Users/terry/Documents/cc_projects/my_agent1/server/services/worker.py:92-99)）：`_claim_lock` 保护了"查询 + 状态更新 + commit"的完整事务，正确防止了多 worker 竞争同一任务。

- **`memory.py` 和 `memory_store.py` 的分层清晰**：`memory.py` 是业务逻辑层（去重、合并、摘要），`memory_store.py` 是 ChromaDB 的薄封装。但 `memory_store.py` 直接导入 `store.py` 的私有函数破坏了模块边界。

- **`MemoryStore` 和 `VectorStore` 共享 ChromaDB 持久化目录**：两者通过不同 collection（`"memories"` vs `"knowledge_base"`）隔离，设计合理但值得在文档中注明。

- **`database.py` 的 `_migrate` 使用 `raw_connection()` 绕过连接池**：仅在启动时调用一次，可接受，但应在代码注释中说明。
